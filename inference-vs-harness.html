Inference vs Harness
Field guide · 2026
A field guide for engineers

Inference is
not a harness.

Most "AI capabilities" people argue about belong to the program around the model — not the model itself. This deck draws the line, and shows you how to redraw it for your own tools.

The confusion
02

You've heard things like

Five sentences,
all subtly wrong.

"Claude has skills now."→ The harness loads skills. The model just reads them.
"GPT can use tools."→ Models emit tool calls. The harness runs them.
"The model remembered our conversation."→ Models are stateless. The harness replayed history.
"ChatGPT decided to search the web."→ The harness exposes a search tool the model can call.
"My agent spawned a subagent."→ Your harness spawned another inference call.
What inference is
03

Inference

A pure function.
Tokens in,
tokens out.

An inference call is one HTTP request. You hand the model a list of messages and a model name. It hands you back the next message. That's the entire surface area.

No memory between calls. No filesystem. No network. No ability to wait, loop, or decide "let me check on that later." It is a function — and like all pure functions, it does exactly one thing.

// the entire model API
// stateless. pure. forgets everything after returning.
function inference(
  messages: Message[],
  tools?: Tool[],
  system?: string,
): Promise<AssistantMessage>

// that's it. that's the model.
What inference cannot do
04

Things the model literally cannot do

If you've ever seen an AI
do any of this — it wasn't the model.

✕Read a file from your disk
✕Run a shell command
✕Remember yesterday's session
✕Call an HTTP endpoint
✕Wait for a build to finish
✕Loop until a condition is met
✕Trigger another model call
✕Enforce a permission policy
✕Compact its own context
✕Schedule a future task

Each of these is a harness responsibility. The model can ask for them — by emitting structured tool calls — but it cannot do them.

What a harness is
05

Harness

The program
around the model.

A harness is everything that turns a stateless function into something that feels alive: a conversation loop, tool execution, file I/O, context management, permission gates, hooks, plugins, and the orchestration of one or many inference calls.

The same model running in two different harnesses behaves like two different products. The model didn't change — the program around it did.

The split
06

A line you can draw with a ruler

What lives where.

  • Loading skills, plugins, MCPcode
  • Hooks (PreToolUse, Stop, …)code
  • Spawning subagentscode
  • Permission gatescode
  • Context compaction & memorycode
  • Streaming, retry, billing, logscode
  • Scheduling, background taskscode

◉ Lives in the model

  • Reading the messages so farweights
  • Following instructionsweights
  • Choosing which tool to callweights
  • Writing the next tokensweights
  • Knowing things from trainingweights
  • Reasoning across given contextweights

Six things. That's the model's whole job.
Everything else is your code.

Where skills live
08

Skills

A skill is a markdown file
that the harness loads.

When you type /pdf, the model does not know what /pdf means. The harness does. The harness reads the skill file, injects it into the next inference call, and the model — now seeing those instructions for the first time — follows them.

USER
Types /pdf merge a.pdf b.pdf
→
HARNESS
Resolves /pdf → reads skills/pdf/SKILL.md
→
HARNESS
Injects skill content + user args into messages
→
MODEL
Sees skill for the first time. Decides which tools to call.
→
HARNESS
Executes Bash, Read, Write. Returns results.

Take the skill file away and the "skill" vanishes. The model's weights are unchanged.

Where agents live
09

Agents & subagents

An "agent" is just a configuration
the harness applies to an inference call.

A subagent is not a separate AI. It is the same model, called by the same SDK — but the harness:

  • Swaps in a different system prompt
  • Restricts the available tools
  • Starts with an empty message history
  • Returns only the final message to the parent loop

That's the entire mechanism. Orchestration — when to spawn, how to merge results, whether to run them in parallel — is harness code. The model has no concept of "I am a subagent."

# a subagent, demystified def spawn_subagent(task, agent_type="explore"): cfg = harness.load_agent(agent_type) # cfg = { system_prompt, allowed_tools, model } result = sdk.messages.create( model=cfg.model, system=cfg.system_prompt, tools=cfg.allowed_tools, messages=[{"role": "user", "content": task}], ) return result.content # parent loop continues
Where plugins live
10

Plugins

Plugins extend the harness,
not the model.

A plugin is a bundle the harness loads at startup. It can register new commands, new tools, new subagents, new hooks, new MCP servers, new keyboard shortcuts. The model never "installs" anything — it just sees the resulting tool list in the next inference call.

0
bytes the model loads
Models do not download, parse, or install plugin code. Ever.
N
tools the harness adds
Plugin tools appear in the tools parameter of inference calls. To the model, they look identical to built-ins.
1
model, unchanged
Swap your model from Opus to Sonnet — every plugin keeps working. They were never tied to weights.
Where hooks live
11

Hooks

Hooks fire on harness events.
The model never sees them.

A hook is a shell command (or function) the harness runs at a specific point in its own lifecycle. The model isn't even told it happened — unless the harness chooses to surface the output as a tool result or a system message.

UserPromptSubmitAfter user hits enter, before inference is called.e.g. block secrets in prompts
PreToolUseModel asked to run a tool. Harness can approve, block, or rewrite.e.g. require approval for rm
PostToolUseTool finished running. Harness has the result in hand.e.g. auto-format edits
NotificationHarness wants user attention (idle, waiting on input).e.g. ping Slack
StopTurn ended. Useful for logging, lint, status updates.e.g. run tests
SubagentStopSubagent returned. Parent loop about to resume.e.g. audit subagent diffs

If you find yourself wishing "the model would always X before doing Y" — that is a hook, in your harness. Not a model retraining job.

The bridge
12

Tools — the only thing crossing the line

Tool calls are how the model
asks the harness to act.

◉ Harness owns
  • Defining tool schemas
  • Exposing them to the model
  • Receiving tool_use blocks
  • Executing them safely
  • Returning tool_result
  • Looping until done
tool_use
tool_result
◉ Model owns
  • Reading the conversation
  • Choosing whether to call a tool
  • Choosing which tool
  • Filling in arguments
  • Reading tool results
  • Deciding what to do next

This is the entire contract. The model proposes. The harness disposes.

Build your own
13

A harness in ~40 lines

It really is this small.

import anthropic, subprocess client = anthropic.Anthropic() TOOLS = [{ "name": "bash", "description": "Run a shell command", "input_schema": {"type":"object", "properties":{"cmd":{"type":"string"}}, "required":["cmd"]} }] messages = [] while True: user = input("> ") messages.append({"role":"user","content":user}) while True: r = client.messages.create( model="claude-opus-4-7", max_tokens=4096, tools=TOOLS, messages=messages, ) messages.append({"role":"assistant","content":r.content}) if r.stop_reason != "tool_use": break results = [] for block in r.content: if block.type == "tool_use": out = subprocess.run(block.input["cmd"], shell=True, capture_output=True, text=True) results.append({"type":"tool_result", "tool_use_id":block.id, "content":out.stdout + out.stderr}) messages.append({"role":"user","content":results}) print(r.content[-1].text)

That file is a complete agentic coding harness. Drop it on a server. Point it at a repo. It will read, write, run commands, debug, and iterate.

Notice what isn't there: no fine-tuning, no special model, no proprietary API. Just messages.create in a loop and a subprocess.run for the tool.

Every "AI coding tool" you have heard of is some elaboration on these 40 lines.

What separates a toy from a tool

From 40 lines to a real harness.

Every layer above the core is something you choose to build. None of it changes the model.

The takeaway
15 · END

Remember this

The model is a function.
Everything you love
about your AI tool
is the harness.

Want different behavior? Don't wait for a new model. Change the harness — add a skill, write a hook, register a tool, swap the system prompt. The lever you're looking for is almost always on your side of the line.
